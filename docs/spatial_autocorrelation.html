3 Spatial Autocorrelation

3.1 Introduction

Aims

The aims of this practical are to:

  1. Understand why spatial autocorrelation needs to be considered when analysing spatial data.
  2. Understand the techniques we can use to assess for global and local spatial autocorrelation, including the underlying mathematics and assumptions.
  3. Test for spatial autocorrelation in ArcGIS and interpret outputs correctly.

Application

To that end, we’ll use county-level data from the 2024 US General election to investigate global and local spatial autocorrelation.

Data

The county-level data for this practical are available in [data/us-census] as follows:

  • US county geometries for 2023, mapped at a 1:500,000 scale [Source]
  • US county-level Presidential results for the 2024 elevation [Source], which includes:
    • Democratic vote total [votes_dem] and proportion (per_dem)
    • Republican vote total [votes_gop] and proportion (per_gop)

Note that other datasets are present in this directory and will be utilised later in the unit.

Tools

Spatial Autocorrelation (Global Moran’s I) (Spatial Statistics Tools), Cluster and Outlier Analysis (Anselin Local Moran’s I) (Spatial Statistics Tools), High/Low Clustering (Getis-Ord General G) (Spatial Statistics Tools), Hot Spot Analysis (Getis-Ord Gi*) (Spatial Statistics Tools), Project (Data Management Tools)

3.2 Practical

[1] Open ArcGIS and load county geometry data [cb_2023_us_county_500k.shp, n=3235]:

Note some context here. The county level is for administrative purposes, but interesting for looking at patterns. The actual state vote (which gives reps to the electoral college) is % based at the state-level.

[2] For simplicity, remove non-contiguous counties using Select By Attributes (STATE_NAME = Alaska, Hawaii, Puerto Rico, Guam, American Samoa, Commonwealth of the Northern Mariana Islands, United States Virgin Islands), and either Delete Rows, or save selected to new features (Data -> Export Features -> Use the selected records -> cb_2023_us_county_500k_contig)

STATE_NAME <> ‘Alaska’ And STATE_NAME <> ‘Hawaii’ And STATE_NAME <> ‘Puerto Rico’ And STATE_NAME <> ‘Guam’ And STATE_NAME <> ‘American Samoa’ And STATE_NAME <> ‘Commonwealth of the Northern Mariana Islands’ And STATE_NAME <> ‘United States Virgin Islands’

[3] Use Project to create new feature layer a using a suitable projected CRS e.g., 102004 Lambert Conformal Conic, rather than the current geographic CRS (NAD83 - EPSG:4269). This will cause issues if measuring geometries (e.g., areas, distances), as discussed in Understanding GIS (Week 2)

[4] Load in presidential results [2024_US_County_Level_Presidential_Results.csv] and join to geometries, using Joins and Relates -> Add Join (NAMELSAD, county_name).

Note an alternative approach would be to use the Federal Information Processing Series (FIPS) code, stored in GEOID and county_fips respectively, but different data formats (Text, Long) means we don’t get any matches, and would have to do some more data processing.

[5] Inspect the Attribute Table for completeness and for key stats i.e., Explore Statistics of per_gop. Mean = 0.66% at a county level.

Question: Why does this not match the overall vote % [i.e., population-excluded]

  • gop_total = 81,382,360 (50.4% of total)
  • dem_total = 77,151,397 (47.8% of total)
  • gop+dem = 158,533,757
  • total_votes = 161,380,528

This is close to the reported results, possibly due to removing non-contiguous states.

Data caveat:

“Although the data in this repository is extensive, it is not considered the authoritative source. Researchers are encouraged to verify specific results from primary data sources when needed”

[6] Modify symbology:

  • Vary symbology by attribute (per GOP), using ‘Unclassed Colors’
  • Colour scheme = Red-Blue (continous)
  • Colour scheme > Properties > Reverse
  • Double click values, update 0-1, same for labels

Note Symbology using ‘Unclassed Colors’ doesn’t always update correctly i.e., sometimes fails to show the data at all. James can you check this?

[7] Run Spatial Autocorrelation (Global Moran’s I):

  • per_GOP as input
  • Generate Report
  • Conceptualization of Spatial Relationships should be familiar following Practical 2, we’ll use Contiguity edges corners here
  • TO DO:
    • Discuss row standardisation: This divides each neighbor’s weight by the total number of neighbors, making the rows sum to 1. This prevents bias caused by some areas having many neighbors while others (like isolated regions) have very few.
    • Communicate the underlying mathematics, as interpretation is very easy with the html output

[8] Interpret initial output:

First, it’s important that we look at any warnings or error messages that might arise. For this tool, we have some:

WARNING 000642: Problems reading 1 of 3109 total records
WARNING 000848: Features with bad records (only includes first 30): us-county-contig.FID = 292.

If we check the Attribute Table, this is for the District of Colombia which has <Null> vote share. This is because of a mismatch between our geometry data (the District of Colombia as a single feature), and our presidential results table, whereas we have the vote share broken down at ward level (Ward 1 - Ward 8). We could either remove this geometry (Q: will this have a significant effect at our scale of analysis?) or we can add the vote share in manually.

WARNING 000846: 4 feature(s) had no neighbors which generally invalidates the statistical properties of a test
WARNING 000847: Features with no neighbors (only includes first 30): us-county-contig.FID = 426, 534, 2018, 2616.

If we check the Attribute Table, this is for four counties with no direct borders:

  • Nantucket County (25019)
  • Dukes County (25007)
  • Richmond County (36085)
  • San Juan (53055)

Note these have no borders when using Contiguity edges corners, but likely would be valid features using a distance-based approach, so we can keep them for now.

The next step is to address the source of those warnings.

[9] Add results for the District of Columbia [Source]:

District of Columbia,11001,District of Columbia,21076,294185,315261,-273109,0.069,0.9331,-0.8641

Open 2024_US_County_Level_Presidential_Results.csv in Excel, create a new row, copy the comma-delimited data above into the first column and split into the relevant columns using Data → Text to Columns → Delimited → Comma → Finish, then save the csv.

Note because we used a dynamic table join before, rather than Join Field, District of Columbia should now have data!

[10] Re-run Spatial Autocorrelation (Global Moran’s I) and inspect outputs:

  • Warnings addressed in [9] no longer present

  • z-score 26.814927
  • p-value 0.000000
  • Clustered: Given the z-score of 26.814927, there is a less than 1% likelihood that this clustered pattern could be the result of random chance.

3.3 Extra

As a comparison, we can run with 2008 election results (Obama vs. McCain), stored in 2008_US_County_Level_Presidential_Results.csv

Link the results to the county geometry data (Add Join) and re-test for global and local spatial autocorrelation.

Question: How have the results changed over that 16 year period? Is voter spatial autocorrelation greater, less, or about the same and what does this reveal?

Question: What is a limitation of our current approach? Hint: Sort the gop_per column for the 2008 results.

Answer: the county geometries / names / FIPS codes might have changed in this period so we have 80 counties of 3109 where there is no matching name. Instead, we should really use time-accurate geometries, available here.
