Chapter 10 Spatial Interaction Modelling

10.1 Introduction

Aims

The aims of this practical are to:

  1. Understand the aims and common applications of spatial interaction (gravity) models.
  2. Understand the design and calibration of spatial interaction models.
  3. Apply spatial interaction models to a contemporary sociological issue.

Application

To that end, we’ll develop spatial interaction models to predict internal migration flows between local authorities in England and Wales.

Data

All data for this practical have been sourced from the Office for National Statistics (ONS) at the local authority level and are found in [data/ons/local-authority] as follows:

  • ONS internal migration flows for local authorities in England and Wales (2021) [Source]
  • ONS district geometries (2021) [Source]
  • ONS gross domestic product (GDP) per head at current market prices [Source]
  • ONS population [Source]

Tools

Feature To Point (Data Management Tools), Generate Near Table (Analysis Tools), Generalized Linear Regression (GeoAnalytics Desktop Tools), Notebooks in ArcGIS Pro

————— Model description

Classic multiplicative gravity model:

(EQ1) \(T_{ij} = k \frac{V_i^μ W_j^α}{d_{ij}^{\beta}}\)

Can also be written from Wilson (1971) as:

(EQ2) \(T_{ij} = kV_i^μ W_j^αd_{ij}^{-\beta}\)

The latter is mathematically identical, but removes the fraction (simpler).

From Dennett:

“The flows between an origin and destination are proportional to the product of the mass of the origin and destination and inversely proportional to the distance between them.

As origin and destination masses increase, flows increase, but as distance increases, flows decrease, and vice versa.”

Here:

\(T_{ij}\) is the predicted flow between origins and destinations. \(V\) is the origin “emissiveness” i.e., origin population and salaries (e.g., ORIG_POP, ORIG_GDP, ORIG_GDP_CAP) \(W\) is the destination “attractiveness” i.e., destination population and salaries (e.g., DEST_POP, DEST_GDP, DEST_GDP_CAP) \(d\) is the cost of flows i.e., distance (NEAR_DIST) \(k\), \(μ\), \(α\), \(\beta\) are model parameters to be estimated

These parameters relate to the scaling effect or importance of the variables they are associated with.

We can start with some basic parameters i.e., effects scale in a linear fashion. For a 1 unit increase in population, we might expect a 1 unit increase in flows of people from that origin.

Newtons gravity uses a power law of \(\beta=-2\), so here are the starting values:

\(k=1\), \(μ=1\), \(α=1\), \(\beta=-2\)

For \(V\) (origin “emissiveness”) we will use origin population (orig_pop) For \(W\) (destination “attractiveness”), we will use destination GDP per capita (dest_gdp) For \(d\), we use \(d^\beta\), where \(d\) = NEAR_DIST and \(\beta=-2\)

gm1 = \(V\) * \(W\) * \(d\)

————— Modelling

[13] Add Field (gm1), Float, following equation (distance in km)

pow(!orig_pop!, 1) * pow(!dest_gdp!, 1) * pow(!NEAR_DIST!, 1000, -2)

This produces an unconstrained output (unitless), as we have not used the k factor, which converts our model to a total constrained version by scaling to match the total number of flows i.e., sum of all observed flows divided by the sum of all of the other elements in the model.

[14] Using Explore Statistics, return the sum of flow (102228926.73000628) and gm1 (3912.65851170132), and then calculate a new field incorporating the scale factor (gm1_scale)

Data = 3698001.8812999986 (# internal migrations) Model = ~63224.55841802744 (sum of gm1)

!gm1! * (3698001.8812999986 / 63224.55841802744)

[!] Note: we can’t do this all via Add Field, as this is a row-wise calculation

[15] Model evaluation

First, check the total of gm1_scale matches the total flow (it does!), so our model is total constrained

[!] However, the output is pretty abysmal here (R2=0.07, Create Chart, Scatter Plot), because we haven’t tuned our parameters!

————— Tuning parameters

If we log both sides of the equation above, we get the following:

(EQ3) \(\mathrm{ln}T_{ij}=k+μ\mathrm{ln}V_i+α\mathrm{ln}W_j−β\mathrm{ln}d_{ij}\)

This is basically a regression model!

However, there are problems with using a log-linear approach, and instead we use Poisson regression, which is better for skewed distributions (like our flow data)

[16] Visualise Statistics - Histrogram - Flow

So our Poisson regression looks like the following:

(EQ4) \(λ_{ij}=\mathrm{exp}(k+μ\mathrm{ln}V_i+α\mathrm{ln}W_j−β\mathrm{ln}d_{ij})\)

[17] First, log the independent variables using Add Field -> math.log(!orig_pop!) -> math.log(!dest_gdp!) -> math.log(!NEAR_DIST! / 1000) # converting to km -> * see section below on math.log()

[18] Run Generalized Linear Regression (GeoAnalytics Desktop Tools) -> Must use GDT version, as this allows for table inputs -> Flow (dependent) -> Logged variables (independent) -> Count (Poisson)

————— Log issues

Python: math.log() uses e as the default

R: log() uses exp(1) as the default, which is identical to above: https://www.rdocumentation.org/packages/base/versions/3.6.2/topics/log

Excel: log() defaults to base10, which is different! Either plug in the above i.e., log(val, EXP(1)), or use LN()

# List maps
m = aprx.listMaps()[0]

# Access table
table = m.listTables("la_near_table_v2")[0]
table_path = table.dataSource
workspace = arcpy.Describe(ta

# List of fields
fields = [f.name for f in arcpy.ListFields(table_path)]

# Data to pd
rows = arcpy.da.SearchCursor(table_path, fields)
df = pd.DataFrame(list(rows), columns=fields)

# Create formula and run glm 
model_formula = "flow_int ~ C(ORIG_LAD21CD) + orig_pop_log + dest_gdp_log + near_dist_log - 1"
model = smf.glm(formula=model_formula, data=df, family=sm.families.Poisson()).fit()
