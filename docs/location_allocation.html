9 Location Allocation

9.1 Introduction

Aims

The aims of this practical are to:

  1. Understand the aims and theory of location allocation models.
  2. Understand the methods and criteria used for location allocation.
  3. Investigate the sensitivity of location allocations to those methods-criteria.

Application

To that end, we’ll use location allocation models to optimise locations for new defibrillators, using population demand and the transport network as inputs.

Our analysis is focused on the Western Isles of Scotland, given links between socioeconomic deprivation, urban-ruralness and defibrillator access (Burgoine et al., 2024). For every minute that someone’s in cardiac arrest without receiving CPR and having a defibrillator used on them, their chance of survival decreases by 10%. Optimising defibrillator locations to minimise response times or maximise coverage could save lives.

Data

  • Defibrillator locations listed on the National Defibrillator Network, managed by the British Heart Foundation and associates (The Circuit), available in [data/bhf] [Source]
  • Ordnance Survey (OS) Open Roads transport network, available in [data/os/open-roads] [Source]
  • UK Centre for Ecology and Hydrology (UKCEH) gridded population at 1 km resolution (2021), available in [data/ukceh] [Source]
  • ONS boundaries data, filtered to the Western Isles, available in [data/ons/national] [Source]

Tools

Create Network Dataset (Network Analyst Tools), Build Network (Network Analyst Tools), Make Location-Allocation Analysis Layer (Network Analyst Tools), Near (Analysis Tools), Snap (Editing Tools), Round Up (Spatial Analyst Tools), Int (Spatial Analyst Tools), Create Feature Dataset (Data Management Tools), Raster to Point (Conversion Tools), Feature Class To Geodatabase (Conversion Tools)

9.2 Practical

9.2.1 Pre-processing

[1] Load road network [road_link_westcoast.shp], population grid [uk_population_2021_1km.tif] and islands [west-coast-islands.shp] shapefile: all BNG, subset to Western Isles.

[2] Load defibrillator location (defibrillator_data.csv), XY Table to Points using lat-long (WGS_1984), and then Project to BNG

[3] Clean defibrillator dataset, keeping valid entries and spatially aligning to network):

  • Use Near to add distance from the defibrillators to the road network (modifies input)
  • Visualise Statistics
  • Select by Attributes (NEAR_DIST > 100) and delete selected rows (n=15) - recording errors?
  • Use Snap to snap points to the network (100 m tolerance, modifies input):
    • Type = Edge
    • Distance = 100
  • Check for validity

Option Subset to defibrillators with 24 hour access?

[4] Clean Demand points

  • Raster to Point (population) to convert raster to Points at cell centroids
  • Use Near to add distance from the centroids to the road network (modifies input)
  • A reasonable distance cutoff would be 700 m due to Pythagoras: sqrt(0.5^2 + 0.5^2) = 0.7071 km
  • Use Select by Attributes to remove Points where distance > 700 m
  • Use Snap, with distance = 700 i.e., raster generalized to 1 km, so we “know” that the population is within a ~700 m radius

An alternative approach would be to use Round Up on the raster, convert to integer via truncation using Int, use Raster to Point to produce centroids, Buffer by 500 m and then use Minimum Bounding Geometry. This would produce a polygon grid where the cell value [grid_code] contains the number of people per 1 km cell.

This seems unnecessarily complicated, given that Raster to Polygon (Conversion Tools) exists, but this produces an inferior output, as neighbouring cells with identical values are merged.

Based on the output above, Create Random Points could be used on a polygon-by-polygon basis, which could then be adjusted to the network using Snap. This approach reproduces the correct number of people within each 1 km grid square, and perhaps a more realistic situation than relying on centroid locations only. However, it is much more complicated, and given the aims of the practical, the former is preferred.

[5] For efficiency, we are going to focus on a single island, so use Clip to subset all datasets to Isle of Mull (Name = Mull). This includes:

  • the road network,
  • population points produced in [4] which represent the “demand” locations
  • current defibrillator locations (n=38)

[6] To run Location Allocation in ArcGIS, we can’t use a simple features layer, we need to use a Feature Dataset, which has stricter rules regarding projection - all layers must share a common coordinate system, unlike a gdb (although it can be stored within a gdb). This can be produced using Create Feature Dataset (output = empty)

Discuss Feature Dataset fully, compare to gdb and shp

[7] We can then import features to the Feature Dataset:

  • Right click in Catalog Pane
  • Import to open Feature Class to Geodatabase
  • Add Road network and run

9.2.2 Network building

[8] Use Create Network Dataset to defines the rules and sources of the network, using a suitable name output e.g., road_model and selecting the Source Feature Class imported in [7]

[9] Run Build Network, which is processing action that enforces the rules defined in [8], maps connectivity, and generates the data structure needed for routing.

[10] Visual inspection of output. We can now use this network dataset for lots of tasks via New Network Analysis Layer e.g., Service Area, Route, Closest Facility, …

Note it is possible to use network datasets provided by ESRI, which allows for quicker calculation of travel time for different modes of travel. However, this does require credits to be used, and is “black-box” for us.

9.2.3 Location Allocation: P-median

i.e., minimize the total distance or time traveled by users to their nearest facility, weighted by the amount of demand or “How do we minimise average distance for everyone?”

[11] Make Location-Allocation Analysis Layer (~5 mins)

  • Network Data Source = road_model
  • Name = la_pmedian_mull
  • Problem Type = “Minimize impedance” (P-median)

[12] Location-Allocation Layer tab: Load current defibrillator locations

  • Input Data group
  • Import Facilities (defib)
  • Property FacilityType = Required i.e., sites always included in the solution

[13] Location-Allocation Layer tab: Add demand points

  • Input Data group
  • Import Demand Points (population)
  • Property Weight = grid_code (the population value from the raster)

[14] First run using current configuration, setting the number of facilities to match the number of features in defib (for Mull, n = 38)

Output:

  • Sum of allocated weight in “Demand Points”: 3050.685136.
  • Sum of allocated weighted impedance in “Demand Points”: 3685274.462097.
  • Sum of valid unallocated weight in “Demand Points”: 0.000000
  • Succeeded at 30 June 2026 14:08:32 (Elapsed Time: 3.30 seconds)

Interpretation:

  • Sum of allocated amd unallocated weights not particularly meaningful, as we are using ‘P-median’ where all demand points are allocated to a facility
  • Sum of allocated weighted impedance is the key analysis metric i.e., the total population-weighted travel cost across all demand points to their assigned nearest facility).
  • This is our baseline for comparison

[15] Second run with some new candidate locations

  • Dissolve road network, otherwise we will generate points for every feature, not the entire network
  • Create Random Points, n=100, random_mull_n100, constrain to dissolved network
  • Add to Location Allocation Layer (FacilityType = Candidate)
  • Solve again where number of facilities = 50 i.e., 38 existing facilities in defib + 12 new locations

Output:

  • Sum of allocated weight in “Demand Points”: 3050.685136.
  • Sum of allocated weighted impedance in “Demand Points”: 2720895.446077.
  • Sum of valid unallocated weight in “Demand Points”: 0.000000

Interpretation:

  • Sum of allocated and unallocated weights identical to previous situation [correct]
  • Sum of allocated weighted impedance has dropped “significantly”
  • Absolute change = -964,379, Percentage reduction = 26.2%
  • Relative change is more meaningful here, as absolute change is measured in “person-meters” (when impedance is metres and weights are population)

[16] Test with an idealised scenario (i.e., no existing defibrillators)

  • Use Create Random Points to create 1000 random points, constrained to the network. An alternative approach would be regular spacing to ensure full coverage.
  • Open Facilities Attribute Table and delete all rows
  • Add random locations, all as Candidate
  • Set number of facilities = number of features in defib (n = 38) i.e., we are now evaluating the optimum locations (based on the p-median approach) for our 38 defibrillators, of the 1000 possible candidates (not all possible locations).

Alternatively we could use Facilities = 50, to facilitate comparison with [15]

Output:

  • Sum of allocated weight in “Demand Points”: 3050.685136.
  • Sum of allocated weighted impedance in “Demand Points”: 2430836.519422.
  • Sum of valid unallocated weight in “Demand Points”: 0.000000.

Interpretation:

  • Sum of allocated weighted impedance (~2430836) is comparable (and better than) to [15, n=50] and significantly less than [14, n=38]
  • Absolute difference = 1,254,438, Percentage reduction = 34%
  • In turn, we can improve access to defibrillators with location optimisation and without increasing the total number!

Average distances are also lower, which can be demonstrated Explore Statistics (although should be weighted by population):

  • Current mean = 2227.343557515982
  • “Optimum” mean = 1815.0352018341623

To finish this section, ensure full understanding of how p-median works. What is it actually evaluating?

9.2.4 Location Allocation: Maximise coverage

i.e., maximize the total amount of demand (or population) covered within a cutoff (e.g., distance threshold)

[17] Make Location-Allocation Analysis layer (~5 mins):

  • Network Data Source = road_model
  • Name = la_maximise_mull
  • Cutoff = 1000 (1 km)
  • This can be justified based on the rapid-response required in cases of Out-of-Hospital Cardiac Arrest (OHCA)
  • Problem Type = “Maximise Coverage”

In ArcGIS, there is the option to switch the problem type for an existing Location-Allocation Analysis Layer i.e., from P-Median to Maximise Coverage. From my testing, this doesn’t work, hence why we’re creating a separate layer. James can you reproduce?

[18] Location-Allocation Layer tab: Load current defibrillator locations:

  • Input Data group
  • Import Facilities (defib)
  • Property FacilityType = Required i.e., current sites always included in the solution

[19] Location-Allocation Layer tab: Add demand points

  • Input Data group
  • Import Demand Points (population)
  • Property Weight = grid_code (the population value from the raster)

[20] First run using current configuration, setting the number of facilities to match the number of features in defib (for Mull, n = 38), this time using maximise coverage.

Output:

  • Sum of allocated weight in “Demand Points”: 2008.805185.
  • Sum of allocated weighted impedance in “Demand Points”: 981885.557754.
  • Sum of valid unallocated weight in “Demand Points”: 1041.879952.

Interpretation:

  • In this case we are interested in the ratio of allocated-unallocated weights
  • Given the selected distance threshold (1 km) and the current distribution of facilities and demands, 1041 people are not served by current distribution
  • Coverage is therefore ~62%

[21] Test with an idealised scenario (i.e., no existing defibrillators)

  • Open Facilities Attribute Table and delete all rows
  • Add random locations created in [16], all as Candidate
  • Set number of facilities to match the number of features in defib (n = 38) i.e., we are now evaluating the optimum locations (based on the maximise coverage approach) for our 38 defibrillators out of the 1000 possible candidates

Output:

  • Sum of allocated weight in “Demand Points”: 2541.885593.
  • Sum of allocated weighted impedance in “Demand Points”: 1222049.489314.
  • Sum of valid unallocated weight in “Demand Points”: 508.799543.

Interpretation:

  • With the same number of facilities, the same method (maximise coverage) and cutoff (1 km), we have increased the population coverage to ~83%
  • Optimisation is valuable!

[22] Evaluate changing cutoff

Cutoff = 5000 m

  • Sum of allocated weight in “Demand Points”: 3050.685136.
  • Sum of allocated weighted impedance in “Demand Points”: 2794852.161580.
  • Sum of valid unallocated weight in “Demand Points”: 0.000000.
  • Coverage = 100%

i.e., when the cutoff is large, all demand points can be connected

Cutoff = 500 m i.e., run for 2.5 minutes to get defib, run back

  • Sum of allocated weight in “Demand Points”: 1957.550792.
  • Sum of allocated weighted impedance in “Demand Points”: 363134.112470.
  • Sum of valid unallocated weight in “Demand Points”: 1093.134344.
  • Coverage = 64%

i.e., reducing the cutoff reduces coverage, but this is still comparable to the baseline, even with a more restrictive cutoff (500 m vs. 1 km)

9.3 Extra

Using the maximum coverage approach, how many facilities do we need to ensure 95% of the population are within 1 km of an AED?

  • Baseline = 62%
  • N=38 (optimised) = 83%
  • N=50: Sum of allocated weight in “Demand Points”: 2672.349411 = 87% …
  • N=100: Sum of allocated weight in “Demand Points”: 2937.332799 = 96%

How do our results compare to other islands? Repeat the analysis for Jura…

What key population metric are we not including?

Answer: non-resident population i.e., tourists!

What are some weaknesses of our approach?

Answers: population generalisation. Using distance as a metric, rather than travel time. Not a full constraints analysis i.e., which locations are actually suitable for AEDs?
